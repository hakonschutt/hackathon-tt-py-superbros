# Superbros Hackathon — TypeScript-to-Python Translation Tool (TT)

## Context

This is a 3-hour AI-assisted competition to build a **TypeScript-to-Python translation tool** (`tt`). The tool translates parts of [Ghostfolio](https://github.com/ghostfolio/ghostfolio) (a wealth management app) from TypeScript to Python. We are scored on **correctness, code quality, and understanding**. The tool MUST NOT use LLMs for translation — only for building the tool itself.

See [COMPETITION_RULES.md](COMPETITION_RULES.md) for full rules and [README.md](README.md) for setup.

## Code Quality Rules

These apply to code in `tt/tt/` (the translator itself):

- Type hints on ALL function signatures
- Docstrings on all public functions and classes
- Use `pathlib.Path` for file paths — never string concatenation
- Proper error handling: specific exceptions, informative messages
- Keep functions focused — single responsibility, max ~30 lines
- Use `logging` or `loguru` for output — minimize raw `print()`
- Follow existing codebase conventions when modifying existing code

## Competition Rules (CRITICAL)

1. **NO LLMs in translation** — `tt` must not call any LLM API during translation
2. **No pre-written domain logic** — `tt/` must not contain project-specific financial calculations
3. **No project-specific mappings in tt/ core** — use `tt_import_map.json` in scaffold dir
4. **Wrapper files are immutable** — `app/main.py` and `app/wrapper/` must be byte-for-byte identical to the example
5. **Translation output goes in `app/implementation/` only** — nothing else may be generated
6. **Frequent commits** — git log must show gradual development
7. **SOLUTION.md** — must explain the approach; judges will ask about it

Run `make detect_rule_breaches` often to catch violations early.

## Architecture of the Translator

The translator (`tt`) works as a pipeline:

1. **Scaffold setup** — copies `ghostfolio_pytx_example/` as base, overlays support modules from `tt/tt/scaffold/`
2. **Translation** — reads TypeScript source files from `projects/ghostfolio/`, applies transformations, writes Python to `translations/ghostfolio_pytx/app/implementation/`
3. **The translated code** must implement the abstract interface in `app/wrapper/portfolio/calculator/portfolio_calculator.py`

### What the scaffold already handles (~48 tests pass with zero translation)

- HTTP wiring, auth, response structure
- Simple cost-basis tracking (BUY adds, SELL subtracts)
- Zero-is-correct for closed positions

### What needs actual translation (~87+ failing tests)

- Chart history with per-date market values
- Net/gross performance calculations
- Time-weighted investment tracking
- Dividend and fee tracking
- Portfolio report evaluation (xRay)

## Orchestration (DEFAULT WORKFLOW)

When given a task, the main session acts as **orchestrator**. Full patterns in [`docs/orchestration.md`](docs/orchestration.md).

**Default behavior — always follow this:**

1. **Read** the full requirements
2. **Decide the strategy** using this decision tree:
   - Multiple independent translation aspects → **Divide & Dispatch** (parallel solver agents in worktrees)
   - Parts with dependencies → **Staged Dispatch** (build foundation first, then parallelize)
   - Single ambiguous problem → **Dual-Solve & Pick** (two solvers, pick-best reviews)
   - Single clear problem → One solver agent in a worktree
3. **Dispatch** solver agents using `Agent(isolation: "worktree")` — run independent agents in parallel
4. **Review** each result before merging (read the diff, run `make evaluate_tt_ghostfolio`)
5. **Merge** completed work into main via `scripts/merge.sh` or manual merge
6. **Evaluate** after each merge — run `make evaluate_tt_ghostfolio` and track test count

**Critical rules for dispatching solvers:**
- Include the FULL requirements in the solver prompt — it has no context from this conversation
- Include relevant TypeScript source code or file paths so the solver knows what to translate
- Each solver should commit its work before finishing
- Merge completed agents immediately — don't wait for all to finish
- After merging, always run `make evaluate_tt_ghostfolio` to track progress

## Claude Code Agents

Custom agents in `.claude/agents/`. Invoke with `@agent-name` in Claude Code.

| Agent | Role | Purpose |
|-------|------|---------|
| `@solver` | Worker | Implementation agent — takes a translation task, builds it, tests, commits |
| `@pick-best` | Judge | Compares two implementations, picks the winner |
| `@explorer` | Research | Analyze TypeScript source or codebase structure |
| `@reviewer` | Quality | Review translator code for correctness and quality |
| `@test-writer` | Quality | Write tests for the translator itself |
| `@debugger` | Fix | Diagnose why specific test cases fail |
| `@polisher` | Polish | Final polish — types, docstrings, code quality |
